# FCT-AI · 职能沟通翻译助手

> 把“听不懂的专业话”翻译成“对方能立刻理解并执行的话”。

FCT-AI 是一个面向跨职能团队沟通的 AI 工具，主要解决这类问题：

- 产品同学说需求，研发觉得不够技术化
- 研发同学说优化，产品觉得听不懂价值
- 运营和管理层沟通时，信息颗粒度不一致

本项目通过多 Agent 协作链，将同一段内容按照目标角色最关心的维度进行重写和补充，帮助团队减少沟通损耗、提高决策效率。

## 1. 这个工具到底做什么

如果把这款工具用一句话说明：

**它是一个“跨角色表达翻译器”，不是简单的词语替换，而是“视角转换 + 信息补全”。**

举个直观例子：

- 原句（产品视角）: “我们要做一个推荐功能，提升转化率。”
- 翻译给开发后，不只会换说法，还会自动补充: 数据来源、算法方向、性能目标、风险点、工期拆分。

同理，开发说“QPS、P99、索引优化”时，翻译给产品会变成“用户体验提升、业务承载能力提升、商业价值影响”。

## 2. 核心功能说明

### 2.1 多角色双向翻译

支持 4 类典型角色之间互译：

- 产品经理
- 开发工程师
- 运营
- 管理层

共 12 种翻译方向，不是模板硬套，而是根据源角色和目标角色动态生成。

### 2.2 智能角色识别

当你不确定“这段话更像谁说的”时，系统会先识别角色并给出置信度。

- 置信度高: 自动采用识别结果
- 置信度中: 给出推荐并允许确认
- 置信度低: 由用户手动选择

这保证了既智能又可控，避免误判后“翻译跑偏”。

### 2.3 多 Agent 协作链

系统不是一次性让同一个模型完成所有任务，而是拆分成 3 个专职 Agent：

```text
用户输入
	-> Agent1: 意图识别（谁在说、在说什么场景）
	-> Agent2: 角色翻译（按目标角色关注点重写）
	-> Agent3: 信息补充（查漏补缺、提示风险）
	-> 最终结果
```

拆分的好处是：准确性更高、可解释性更好、后续维护更容易。

### 2.4 流式输出与思考展示

- 翻译内容会实时输出，不用等全部完成
- 支持查看思考过程折叠块（便于理解 AI 生成逻辑）
- 可用于评审“为什么这样翻译”

### 2.5 主动补全

如果原始输入缺少目标角色最关心的信息，系统会在合理范围内补充建议，例如：

- 技术风险
- 验收指标
- 资源与排期建议
- 业务影响范围

## 3. 快速开始（零基础可操作）

下面按“拿到代码后 10 分钟跑起来”的顺序写。

### 3.1 环境准备

你需要先安装：

- Node.js `>= 18`
- npm `>= 9`
- 一个可用的火山引擎 ARK API Key（用于调用 Doubao 模型）

检查本机版本（可选，但推荐）：

```bash
node -v
npm -v
```

### 3.2 安装依赖

在项目根目录执行：

```bash
npm install
```

这条命令会安装前端和后端运行所需的所有依赖。

### 3.3 配置 API Key

1. 进入火山引擎 ARK 控制台: https://console.volcengine.com/ark
2. 开通模型推理相关服务
3. 创建 API Key
4. 确认你已开通模型 `doubao-seed-2.0-pro`

在项目根目录执行：

```bash
cp .env.example .env
```

然后打开 `.env` 文件，填入你的密钥：

```env
ARK_API_KEY=your_api_key_here
```

注意事项：

- 不要在公开仓库提交 `.env`
- 密钥泄露后请及时在平台重置

### 3.4 运行项目

在项目根目录执行：

```bash
npm run dev
```

启动成功后：

- 前端地址: http://localhost:3000
- 后端地址: http://localhost:3001

你只需要打开前端地址即可使用。

### 3.5 使用流程（第一次使用建议）

1. 打开页面后，先选择发言者角色和目标角色
2. 粘贴一段真实沟通内容（或者直接点击选择示例）
3. 点击翻译，观察实时输出（点击翻译前可以选择是否开启思考模式）
4. 查看“补充建议”和“风险提示”
5. 把结果复制到实际沟通场景中验证

## 4. 测试用例（含预期翻译效果）

以下示例可直接复制到页面进行测试。

### 用例 1: 产品视角 -> 开发视角

输入内容：

```text
我们希望在首页增加“猜你喜欢”，根据用户最近浏览和购买行为推荐商品，目标是提高加购率和下单转化。
```

选择：

- 发言者角色: 产品经理
- 翻译给: 开发工程师

预期翻译效果（建议至少覆盖以下点）：

- 需求目标可量化: 例如加购率提升目标、转化率提升目标
- 推荐策略建议: 协同过滤/内容召回/热门兜底
- 数据方案: 埋点字段、日志存储、特征更新频率
- 性能指标: 推荐接口延迟、缓存策略、峰值并发
- 研发排期: 需求拆分到模块级别（召回、排序、接口、埋点）
- 风险提示: 冷启动、脏数据、实验效果不稳定

你会看到输出不再是“业务愿景句子”，而是接近研发可执行的任务描述。

### 用例 2: 开发视角 -> 产品视角

输入内容：

```text
本周我们对订单查询链路做了优化：把热点查询迁移到缓存层，并重写了慢 SQL。核心接口 QPS 从 1200 提升到 1800，P99 延迟从 240ms 降到 90ms。
```

选择：

- 发言者角色: 开发工程师
- 翻译给: 产品经理

预期翻译效果（建议至少覆盖以下点）：

- 用户体验影响: 页面加载更快、卡顿减少
- 业务承载能力: 高峰期能支持更多并发用户
- 商业结果关联: 大促期间更稳定，减少订单流失风险
- 成本影响: 资源利用率更高，潜在服务器成本下降
- 产品规划意义: 为后续大功能发布预留性能空间

你会看到输出从“技术指标汇报”变成“业务价值说明”。

## 5. 提示词设计说明（为什么 AI 能理解角色差异）

这一部分回答两个核心问题：

- 如何让 AI 理解“产品视角”和“开发视角”的差异？
- 提示词中哪些设计点最关键？

### 5.1 角色差异建模方法

在提示词层面，不是简单的只写“你是产品/开发”，而是给出可执行的判定与表达规则：

- 术语特征: 例如“转化率、用户价值”更偏产品，“QPS、延迟、索引”更偏研发
- 关注维度: 每个角色固定关心什么信息
- 输出格式: 不同角色使用不同信息结构（目标、指标、行动项、风险）

这样做的结果是，模型不会只做“同义词替换”，而是进行“信息重组”。

### 5.2 三层 Prompt 职责分离

#### Agent1（意图识别 Prompt）

- 目标: 判断谁在说话、说话场景是什么
- 关键设计: 强制 JSON 结构输出（便于程序稳定解析）
- 价值: 给后续翻译提供准确上下文，减少误翻译

#### Agent2（翻译 Prompt）

- 目标: 按目标角色可理解的方式重写内容
- 关键设计: 每种目标角色有明确输出模板与必答维度
- 价值: 同一内容翻给不同对象时，表达结构自动变化

#### Agent3（补充 Prompt）

- 目标: 补齐信息缺口并提示潜在风险
- 关键设计: 要求从完整性、可执行性、风险性三个角度检查
- 价值: 让结果不仅“听得懂”，还“拿来就能推进工作”

### 5.3 关键设计点总结

1. 角色词典 + 场景词典，降低角色识别歧义
2. 结构化输出约束，提升工程可控性
3. 目标角色导向模板，确保翻译结果可用
4. 主动补全机制，减少来回追问
5. 置信度降级策略，避免 AI 误判造成错误决策

## 6. 系统架构与技术实现（简版）

### 6.1 架构流程

```text
前端（Next.js）
	-> SSE 请求
后端（Express）
	-> Agent Chain 编排
	-> Doubao 模型调用
	-> 流式返回
前端实时渲染结果
```

### 6.2 技术栈

| 层级 | 技术 |
|------|------|
| 前端 | Next.js 15, Tailwind CSS v4, shadcn/ui |
| 后端 | Node.js, Express, TypeScript |
| AI 模型 | Doubao-Seed-2.0-pro（火山引擎 ARK API） |
| 通信方式 | Server-Sent Events (SSE) |
| SDK | OpenAI SDK（兼容模式） |

## 7. 常见问题（FAQ）

### 7.1 为什么我启动后页面没有结果？

优先检查：

- `.env` 中 `ARK_API_KEY` 是否填写正确
- API Key 是否有权限访问 `doubao-seed-2.0-pro`
- 后端是否正常运行在 `3001` 端口

### 7.2 识别角色不准怎么办？

可直接手动选择发言者角色。系统已支持“智能识别 + 人工兜底”机制。

### 7.3 这是翻译工具还是写作工具？

本质上是“角色沟通翻译工具”，强调跨职能可理解性，不追求文学化润色。
